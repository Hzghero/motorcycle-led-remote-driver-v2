## 2. 产品概述

本产品为可遥控的摩托车远近光 LED 驱动器，用于驱动双灯（远光、近光）射灯系统。远光和近光均为 30W，可通过 2.4G 无线遥控器（STM32C011F6P6TR + XL2400T 模块）或原车灯光/喇叭信号进行控制。两灯支持多种交互灯光效果，包括呼吸灯、同步闪、交替闪等。

**主要特性**：
- 远光 30W + 近光 30W 双路独立控制
- 2.4G 无线遥控（发射端 STM32C011F6P6TR + XL2400T（TX only），接收端 STM32C011F6P6TR + XL2400T（RX only）），支持对码配对
- 三档亮度调节（30% / 70% / 100%）
- 七种灯光模式（含呼吸灯、同步闪、交替闪等）
- 可与原车远光、喇叭联动
- 适用于摩托车 12V 电气系统

**主要模块**：
- **主控**：STM32C011F6P6TR（20-TSSOP，18 I/O）
- **2.4G 接收**：XL2400T（接收端单向接收，与主控通过 RF_CSN/RF_SCK/RF_DATA 三 IO 通讯）
- **LED 驱动**：LM3409 × 2，采用 `PWM_GLOBAL + MOS` 架构：主控 1 路全局 PWM 统一调光，4 路 MOS 负责通道导通选择
- **车辆接口**：电源、ACC、黄线（远光）、蓝线（喇叭）、天线

### 4.2 LED 恒流驱动（LM3409）

- **芯片**：LM3409，每路灯独立一颗
- **PWM 与开关复用**：同一路控制信号
  - 恒高电平 → 100% 电流（全亮）
  - 恒低电平 → 关断（熄灯）
  - PWM 占空比 → 对应百分比的电流（亮度）
- **主控输出**：`1 路 PWM_GLOBAL + 4 路 MOS`
- **实现方式**：亮度/呼吸等模拟调光由 `PWM_GLOBAL` 占空比实现；左右远近光通道导通由 MOS 位图控制

### 4.3 与《04-双灯头全局PWM与MOS硬开关功能说明》对齐约束（新增）

> 本节用于“承前启后”：在不破坏本说明书用户语义的前提下，明确新硬件控制边界，指导后续程序演进。

- **用户可见模式集**仍以本《01-功能说明书》为准。
- 《04》仅定义**双灯头硬件控制架构与控制边界**，不单独新增或改写用户侧模式名称与交互语义。
- 后续进入双灯头四通道实现时，采用：
  - **PWM_GLOBAL（1 路）**：统一接入两颗 LM3409 的 PWM 引脚，负责全局亮度与全局关断；
  - **GPIO/MOS（4 路）**：仅负责通道导通路径选择（左远/左近/右远/右近）。
- **关键原则（必须保持）**：呼吸等模拟调光效果仍通过 PWM_GLOBAL 调占空比实现，GPIO/MOS 仅作通道选择，不承担亮度调制职责。

## 5. 无线遥控

### 5.1 遥控器结构

- **三按键设计**
- **左键（锁图标）**：基础灯光模式切换
- **中键**：亮度调节，每次按下循环切换 30% → 70% → 100%
- **右键（电源/波形图标）**：特效模式切换
- 内置指示灯（对码时红色常亮）
- 电池：CR2032 纽扣电池，约 1 年寿命

### 5.2 亮度调节

- 中键按下时，亮度在 **30% / 70% / 100%** 三档循环
- 亮度通过调节恒流驱动的 PWM 占空比实现
- 当前亮度档位对所有灯光模式均有效（常亮、呼吸灯、闪烁等均受其约束）
- PWM 约束（当前实现）：
  - **30% 档 / 70% 档**：输出 PWM 频率约 **20kHz**，占空比分别为 30% / 70%
  - 当前 `PWM_GLOBAL` 物理输出已迁移到 `PA1 / TIM1_CH2`（用于给后续硬件联调对针脚）
  - **100% 档**：输出恒高电平（全亮），不再输出 20kHz PWM
- 在特效模式下，亮灭节奏（如 500ms 高电平 / 500ms 低电平）叠加在上述亮度驱动规则上：
  - 亮相位：按当前亮度档位输出（30/70 为 20kHz PWM，100 为恒高）
  - 灭相位：输出低电平（关断）

### 5.3 2.4G 射频方案（发射端 STM32 + XL2400T / 接收端 XL2400T）

| 项目 | 说明 |
|------|------|
| 芯片 | 发射端：STM32C011F6P6TR + XL2400T（TX only，遥控端）；接收端：STM32C011F6P6TR + XL2400T（RX only） |
| 通信方式 | **单向**：遥控器仅发射，接收器仅接收 |
| 遥控器 | TX only，无需接收功能 |
| 接收器 | RX only；与主控通过 RF_CSN/RF_SCK/RF_DATA 三 IO 通讯 |
| 频率 | 2.4GHz |
| 对码 | 支持一对一对码，更换遥控器或驱动器后可重新配对 |

**改动备注**：发射端无线方案由集成式 XL2417D 调整为“STM32C011F6P6TR + XL2400T（TX only）”，目的是复用 STM32 生态与已验证成功案例，固件与对码流程按接收端 XL2400T 的实现思路落地。另：为实现 STANDBY 三键唤醒，TX 引脚已重映射（左键=PB6/WKUP3，中键=PA0/WKUP1，右键=PA2/WKUP4，PWM_GLOBAL 迁移到 PA1/TIM1_CH2）。

## 6. 灯光模式详细说明

**说明**：以下各模式的亮度均受当前亮度档位（30% / 70% / 100%）约束，由中键调节。

### 6.1 基础模式（左键控制）

| 操作 | 效果（当前固件） |
|------|------|
| 单击 | 近光/远光切换（全灭时单击左键先进入近光） |
| 双击或长按（约 1 秒） | 在“近光 ↔ 远近同亮”之间循环切换（不触发熄灯） |

 
### 6.2 特效模式（右键控制）

| 操作 | 效果（当前固件） |
|------|------|
| 单击 | 进入/切换特效模式（4→5→6→7 循环） |
| 双击或长按（约 1 秒） | 触发全灭（OFF）；但在已全灭状态下该触发无效 |

> 说明：从特效模式回到基础模式时，默认回到“近光开”。
> 备注（2026-04-01）：旧版单灯头硬件仅有单侧远光灯珠，无法实现“左右轮闪”；新版双灯头硬件具备左右远光后可实现该效果。当前双灯头程序已实现模式7左右轮闪，参数可继续调优。

### 6.3 熄灯

- 当前固件的熄灯触发规则：
  - **右键双击/长按**：触发全灭（OFF）；
  - **右键在全灭状态下双击/长按**：无效；
  - **左键双击/长按**：不触发熄灯，仅用于“近光↔同亮”循环；若当前为全灭，则拉起到基础模式1（近光）。
  - **中键双击/长按**：不触发熄灯，等效为中键单击（亮度档位循环）。
- 熄灯后再次点亮：
  - **右键单击**：进入特效模式链路（从模式4起）；
  - **左键单击/双击/长按**：进入基础模式“近光开”（模式1）；
  - **中键单击/双击/长按**：作为亮度操作处理；若当前为全灭，按当前固件实现会拉起到基础模式1后再按亮度规则生效。

| 项目 | 修改前 | 修改后（当前固件） |

| 中键双击/长按（非 OFF） | 触发熄灯（OFF） | 不再熄灯，统一等效为中键单击（亮度循环） |
| 左键双击/长按（OFF） | 未稳定实现拉起 | 明确可从 OFF 拉起到基础模式1（近光） |

| 模式5语义 | 远近交替（历史描述较泛） | 双近光 ↔ 双远光 交替（左右同相） |


### 6.3.2 STANDBY 唤醒后的按键补偿规则（TX）

> 目的：避免“按键只唤醒不执行业务”，并保持单击/长按语义一致。

- 当 TX 从 STANDBY 唤醒时，固件读取唤醒来源标志：`WUF1/WUF3/WUF4`；
- 对应关系：
  - `WUF1 -> M_KEY`（中键）
  - `WUF3 -> L_KEY`（左键）
  - `WUF4 -> R_KEY`（右键）
- 若识别到某一路唤醒，则挂起一次“对应按键单击补偿”；
- 在按键扫描阶段执行补偿前，会再判断该键是否仍保持按下：
  - **已释放**：执行单击补偿（首按即生效）；
  - **仍按下**：跳过补偿，交由长按识别路径处理，避免“单击+长按”双触发。

### 6.3.3 RX 三线输入与遥控命令冲突优先级（正式规则）

> 本节用于固化接收端（RX）在“白线/黄线/蓝线 + 遥控命令”并发下的行为，避免联调歧义。

优先级由高到低如下：

1. **P0：ACC 门控（白线）**
   - `ACC=0`：强制 `OFF`（全灭），覆盖所有输入与遥控命令；
   - 防抖：白线参与统一三线防抖，当前防抖时间 `RX_WIRE_DEBOUNCE_MS=20ms`。

2. **P1：喇叭触发（蓝线）**
   - `HORN=1`：强制爆闪（当前映射模式4）。
   - 进入爆闪时会快照触发前灯态；
   - 防抖：蓝线参与统一三线防抖，当前防抖时间 `RX_WIRE_DEBOUNCE_MS=20ms`。

3. **P2：喇叭释放恢复**
   - `HORN` 从高到低：
     - 若喇叭激活期间收到遥控命令（pending 有效），优先恢复到该遥控目标态；
     - 否则恢复到喇叭触发前快照态。

4. **P3：原车远光同步（黄线）**
   - 仅在 `ACC=1` 且 `HORN=0` 时生效；
   - 首次启用门限：系统进入可用态后，黄线需在窗口内完成一次“双触发”（两次边沿）后才解锁生效；
   - 解锁后：`HI` 上升沿切远光、`HI` 下降沿切近光；
   - 防抖：黄线参与统一三线防抖，当前防抖时间 `RX_WIRE_DEBOUNCE_MS=20ms`；
   - 极性兼容：支持正逻辑与负逻辑配置，双触发与边沿判定均按配置后的“有效电平语义”工作。

5. **P4：遥控命令**
   - 仅在未触发 P0/P1 强制态时直接生效；
   - 若 `HORN=1`，遥控命令缓存为 pending，待 P2 时优先恢复。

### 6.3.3 RX 上电行为（正式规则）

- RX 上电默认安全态：**全灭（OFF）**；
- 不允许上电默认点亮近光/远光；
- 需等待遥控命令或车辆线输入触发后，才进入相应业务灯态。

| `R_KEY`（右键） | `PA2` | STANDBY 唤醒源：`WKUP4` |
| `PWM_GLOBAL` | `PA1 / TIM1_CH2` | 全局亮度与模拟调光输出 |
| `LED`（状态灯） | `PB7` | 仅状态提示，不参与业务灯效 |

> 注：本次重映射后，`PA2` 从 `PWM_GLOBAL` 迁移为 `R_KEY/WKUP4`；`PWM_GLOBAL` 迁移到 `PA1/TIM1_CH2`。

### 7.5 IO 命名规范与角色隔离约束（新增）

为避免后续双灯头改造中“文档名、CubeMX 标签、代码宏名”三者漂移，统一约束如下：

1. **命名三层一致**：
   - 文档功能名（如 `PWM_GLOBAL` / `MOS_L_HI`）
   - CubeMX GPIO Label（同名）
   - `main.h` 宏（同名后缀 `_Pin/_GPIO_Port`）

2. **角色隔离（TX / RX）**：
   - TX 角色：按键扫描、发包、状态指示；
   - RX 角色：解包、灯光执行、低功耗收敛；
   - 业务灯光执行相关 IO（PWM/MOS）仅允许 RX 角色控制，禁止 TX 角色写入。

3. **状态灯与业务灯分离**：
   - `STATUS_LED` 仅用于提示（对码/按键/故障）；
   - `PWM_GLOBAL` 与 `MOS_*` 仅用于灯光业务输出；
   - 禁止复用 `STATUS_LED` 承担业务通道导通控制。

4. **低功耗安全态统一入口**：
   - 所有熄灯/超时/待机入口最终收敛到：`PWM_GLOBAL=0% + MOS 全关`；
   - 不允许出现“仅关 PWM、不关 MOS”或“仅关 MOS、不关 PWM”的分叉实现。

5. **变更同步要求**：
   - 任一 IO 重命名或复用调整，必须同步更新：
     1) 本文档第 7 章接口定义；
     2) 《04-双灯头全局PWM与MOS硬开关功能说明》对应章节；
     3) 工程 `.ioc`、`main.h`、`main.c` 注释。

## 8. 车辆集成

### 8.1 黄线（同步原车远近光）

- 黄线接原车**远光正极**后，可同步原车开关
- **有超车键**：连续拨 3 次超车键 → 激活射灯近光长亮
- **无超车键**：连续拨 3 次远近光开关 → 激活射灯近光长亮

### 8.2 蓝线（喇叭爆闪）

- 蓝线接喇叭正极
- 按喇叭时，射灯进入爆闪模式

### 8.3 白线（ACC 熄灯）

- 正确接好白线后，关钥匙门应能熄灯
- 若将白线接至大灯正极，可实现关原车灯时射灯同时熄灯

### 8.4 负控车型

- 若原车为负控（负触发），黄线、蓝线需接负控转换器，否则相关功能无法实现
- 电动车/电摩接线请另行确认

## 9. 对码流程

### 9.1 对码操作步骤（按当前固件）

更换遥控器或驱动器后，按以下流程：

1. 接收端（RX）上电后自动进入 **5 秒对码窗口**；
2. 发射端（TX）需 **左键+右键同时按下且按下时间差≤150ms**，并持续 **2 秒** 才进入对码发送；
3. 进入对码发送后，TX 发送“对码专用标记包”（非普通业务包）；
4. RX 在窗口内仅接受“对码专用标记包”触发配对；
5. 对码结果反馈：
   - 成功：近光常亮 1 秒后熄灭；
   - 失败：近光快闪 3 下后恢复。

### 9.2 对码机制前后对照（2026-04-05）

| 序号 | 项目 | 修改前 | 修改后（当前固件） |
|------|------|--------|--------------------|
| 1 | TX 进入对码条件 | 左右键长按约 5 秒 | 左右键“同时按下窗口≤150ms”且长按 2 秒 |
| 2 | RX 上电对码窗口 | 描述不一致（依赖操作步骤触发） | 每次上电自动进入 5 秒窗口 |
| 3 | 窗口内配对触发 | 合法业务包也可能触发配对 | 仅“对码专用标记包”可触发配对 |
| 4 | 对码成功反馈 | 远光闪 2~3 次（历史描述） | 近光常亮 1 秒后熄灭 |
| 5 | 对码失败反馈 | 未明确 | 近光快闪 3 下 |
| 6 | 反馈期间业务覆盖 | 成功后可能立即进入业务导致反馈不明显 | 对码反馈窗口独占，结束后恢复业务 |

### 9.3 遥控器 ID 方案（首次随机 + 存储）

| 项目 | 说明 |
|------|------|
| 方案 | 首次随机生成 + 非易失存储 |
| 首次上电 | 遥控器 MCU 生成 32 bit 随机 ID，写入 Flash/EEPROM |
| 之后上电 | 从 Flash/EEPROM 读取已存储的 ID，不再重新生成 |
| 优点 | 无需工厂烧录；换电池后无需重新对码；碰撞概率极低（32 bit 空间约 43 亿） |
| 随机源 | 建议使用 ADC 噪声 + 定时器、或硬件 RNG（若有）作为种子 |

## 11. 时钟与定时基准（固件约束）

- MCU 时钟源：**外部 24MHz 晶振直连**（当前工程不使用 PLL 倍频）。
- 计时相关参数（如 TIM14 节拍、PWM 频率、延时估算）均以 **24MHz** 为基准。
- 若后续改动系统时钟，必须同步复核：
  - TIM1 PWM 频率与占空比映射；
  - TIM14（或其他基础定时器）节拍参数；
  - 与 `HAL_GetTick()` 相关的模式时序常量。

> 说明：该项为本项目关键硬件约束，调参或移植时优先检查。

## 12. 新旧控制架构差异对照（用于后续程序修改）

> 目的：在功能语义不变前提下，明确从“单灯头双通道”到“双灯头四通道”的代码改造方向，降低后续改动断层风险。

| 维度 | 当前基线（01） | 新架构约束（04） | 代码承接建议 |
|------|----------------|------------------|--------------|
| 用户模式语义 | 以 7 种模式与三键交互为准 | 不变 | 保持模式枚举语义稳定，优先做底层重映射 |
| 亮度控制入口 | 两路 PWM 分别驱动 | 一路 PWM_GLOBAL 统一驱动 | 抽象 `Brightness_Set()`，内部改为单入口占空比下发 |
| 通道开关方式 | 主要依赖 PWM 时序 | GPIO + MOS 硬开关 | 抽象 `Channel_ApplyMask()`，将相位切换改为位图翻转 |
| 呼吸/模拟调光 | PWM 占空比实现 | 仍必须由 PWM_GLOBAL 实现 | 禁止把呼吸曲线搬到 GPIO 翻转逻辑 |
| 交替闪实现 | 软相位控制 | A/B 组合切换（如 左远+右近 / 右远+左近） | 状态机改为“相位表驱动”，减少分支硬编码 |
| 低功耗关断 | 分散逻辑 | PWM_GLOBAL=0% + MOS 全关统一收敛 | 建立 `Enter_LowPower_SafeState()` 单入口 |

### 12.1 程序修改“承前启后”实施顺序（建议）

1. **先稳语义**：冻结现有用户模式编号、按键语义、亮度档位行为。
2. **再抽象接口**：先在现有工程抽象 `Brightness_Set / Channel_ApplyMask / Mode_ApplyPhase`，保证外部调用不变。
3. **最后替换底层**：将原双 PWM 分发替换为 PWM_GLOBAL + MOS 位图执行。
4. **低功耗收敛**：统一所有熄灯/超时/待机路径到“PWM_GLOBAL=0% + MOS 全关”。
5. **回归验证顺序**：先基础常亮，再特效，再低功耗，再异常恢复。

### 12.2 差异变更记录要求（文档与代码同步）

- 每次涉及模式行为、时序、占空比映射调整时，必须同步更新：
  1) 本《01-功能说明书》对应章节；
  2) 《04-双灯头全局PWM与MOS硬开关功能说明》边界章节；
  3) 提交记录中的“变更原因 + 对用户可见影响”。
- 建议在提交说明中固定包含：
  - **保持不变**：哪些用户行为保持一致；
  - **内部变化**：底层控制从何到何；
  - **验证结果**：至少 1 条台架结论。
